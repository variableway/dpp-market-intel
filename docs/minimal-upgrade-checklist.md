# Minimal Upgrade Checklist

## 目标

本清单不是“大而全重构”，而是把当前项目从“可演示原型”提升到“可持续维护的准生产原型”。

原则：

- 不推翻现有架构
- 不大改前端页面
- 优先补齐数据、更新、质量控制

## 第一优先级：数据可信度

### 1. 给 dashboard 数据增加来源和更新时间字段

目标：

- 每份数据明确显示更新时间
- 每个关键模块可追踪来源

为什么优先：

- 这是可信度最直接的补强
- 对管理层和后续使用者都非常重要

### 2. 把海关缺口数据补齐成明确任务清单

目标：

- 将当前 `queryChecklist` 变成可执行的补拉任务表
- 明确每个品类的来源、频率、负责人和当前状态

为什么优先：

- 当前最大准确度短板就在海关与 HS 数据缺口

### 3. 区分“估算数据”和“官方数据”

目标：

- 在 JSON 和页面上明确标注哪些字段是估算值，哪些是官方值

为什么优先：

- 能显著降低误用风险

## 第二优先级：更新自动化

### 4. 给静态数据导出增加统一脚本入口

目标：

- 一条命令完成数据准备、JSON 导出、静态构建

建议形式：

- `make build-static`
- 或根目录 `scripts/build_static.sh`

为什么优先：

- 当前步骤已经清楚，但还不够“一键化”

### 5. 给 SQLite provider 增加数据版本表

目标：

- 记录每次导入时间
- 记录导入来源
- 记录导入批次号

为什么优先：

- 这会让 SQLite 从“过渡镜像”提升为“可管理中间层”

### 6. 给 GitHub Actions 增加定时构建

目标：

- 每周或每天定时刷新静态站

为什么优先：

- 可以显著缓解静态数据过期问题

## 第三优先级：新闻质量

### 7. 给 RSS provider 增加去重规则

目标：

- 防止同一条新闻多次进入结果

建议规则：

- `title + normalized_url`
- 或 `source + publishedAt + title`

### 8. 给 RSS provider 增加来源白名单

目标：

- 只接受可信官方源和重点机构源

为什么优先：

- 新闻质量比新闻数量更重要

### 9. 补“业务影响”生成机制

目标：

- RSS 新闻进入系统后，至少生成基础业务影响文本

为什么优先：

- 当前 News 页面最大的价值不在新闻本身，而在“对业务的解释”

## 第四优先级：前端可用性

### 10. 给 Dashboard 增加数据状态提示

目标：

- 明确显示“数据更新时间”
- 明确显示“估算/官方/缺失”状态

为什么优先：

- 小改动，高价值

### 11. 给品类表增加基础筛选

目标：

- 按阶段、是否已有海关数量、是否高优先级补拉进行过滤

为什么优先：

- 可以让页面更像工具，而不只是展示板

## 建议执行顺序

最小推荐顺序：

1. 数据来源和更新时间标识
2. SQLite 版本表
3. RSS 去重 + 来源白名单
4. GitHub Actions 定时刷新
5. Dashboard 数据状态提示
6. 品类筛选

## 阶段完成标准

如果完成以下 5 件事，就可以认为项目从“演示原型”进入“准生产原型”：

1. 所有关键数据有更新时间
2. 估算值与官方值清晰区分
3. SQLite 具备版本记录
4. News 有基本去重和来源控制
5. 静态站支持定时自动刷新
